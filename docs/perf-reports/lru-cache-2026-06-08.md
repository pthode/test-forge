## Finding detail

### R12 — O(1) amortized complexity verification (structural)

The spec (R12, §TD-2) claims `get` and `put` are amortized O(1). Each operation in the implementation was traced:

**`get` (cache.py:70–91):**
- `cache.py:86` — `key in self._entries`: O(1) hash probe on `OrderedDict`.
- `cache.py:87` — `self._entries.move_to_end(key)`: O(1) doubly-linked-list splice in CPython's C implementation of `OrderedDict`. No scan over entries.
- `cache.py:89` — `self._entries[key]`: O(1) hash lookup.

**`put` (cache.py:93–117):**
- `cache.py:110` — `key in self._entries`: O(1) hash probe.
- `cache.py:111` — `self._entries[key] = value` (existing key path): O(1) hash write; `OrderedDict` preserves insertion order on assignment to an existing key — does not reorder, so the `move_to_end` call on cache.py:112 is required and correct.
- `cache.py:112` — `move_to_end(key)`: O(1) splice.
- `cache.py:114` — `len(self._entries)`: O(1) in CPython (length is a cached attribute on dict/OrderedDict).
- `cache.py:115` — `popitem(last=False)`: O(1) removal of the head node of the doubly-linked list.
- `cache.py:117` — `self._entries[key] = value` (new key path): O(1) amortized hash table insert; new keys insert at the tail (MRU end) by OrderedDict semantics.

**`stats` (cache.py:119–132):**
- `cache.py:131` — `len(self._entries)`: O(1). No iteration.

There are no loops, no list searches, no full-table scans anywhere in the three public methods. The R12 claim is structurally confirmed.

### Unbounded loops/recursion check

The only iteration that could be driven by caller-controlled input is the growth of `self._entries`. This is hard-bounded by `self._capacity` (validated >= 1 at cache.py:62–63; the store is capped via the eviction path at cache.py:114–116 before any new insert). The store never transiently exceeds `capacity` entries (spec TD-3). No unbounded iteration exists.

### Blocking / async context check

The module imports only `collections.OrderedDict`, `dataclasses.dataclass`, and `typing` types (cache.py:8–10). No file I/O, no network I/O, no `subprocess`, no `time.sleep`, no CPU-heavy computation. The class is safe to call from any async context.

## Out-of-scope notes (suspicious but not filed)

- **Double hash lookup on the `get` hit path** (cache.py:86 `__contains__` then cache.py:89 `__getitem__`): both are O(1), total cost is 2×O(1) = O(1). No measurable defect. Already logged as B-009.
- **`stats()` allocates a frozen dataclass on every call**: one small object per call, no copies of the entry store, O(1). Not a defect; worth noting that callers in a very tight loop may want to cache the result, but this is a caller pattern issue, not an implementation defect, and no budget ceiling exists to breach.
- **Memory ceiling**: `capacity` bounds entry *count*, not bytes. A caller storing large values can use unbounded memory. This is explicitly documented in spec R10, B-010 already tracks the docstring gap. No new finding.
