## Targeted analysis of the raised concerns

### C1 — Is capacity a hard ceiling on entry count? (memory-bound enforcement)
Result: YES, enforced. No path grows size past capacity.

- put new-key path (cache.py:71-74): when len(self._store) >= self._capacity, it evicts exactly one LRU entry (popitem(last=False)) and bumps _evictions, then inserts the new key — net entry count stays at capacity. The guard is >=, not ==, so even if the store were somehow over capacity it would still evict before inserting (defence-in-depth correct).
- put overwrite path (cache.py:67-70): updates value in place and move_to_end; entry count unchanged.
- __init__ rejects capacity < 1 with ValueError (cache.py:36-37), so the ceiling is always a positive integer; no zero/negative ceiling that would degenerate.
- size is derived as len(self._store) (cache.py:139-141), not a separately maintained counter, so it cannot drift above the real entry count.

Caveat (not a defect): the ceiling bounds entry count, not memory bytes. Arbitrary-sized values mean total memory is unbounded by a fixed factor of capacity * sizeof(largest value). This is the documented contract (spec 3: "No size/weight-based eviction") and an explicit non-goal, not a vulnerability. A caller storing attacker-controlled large values is responsible for bounding value size — worth a one-line note in the docstring (see F1 / BACKLOG).

### C2 — Unhashable / adversarial key handling (state-integrity under failure)
Result: SAFE. A TypeError from an unhashable key never leaves the cache in a partially-mutated state.

In every key-accepting method the very first operation is the "key in self._store" membership test, which is where an unhashable key (e.g. a list or set) raises TypeError:
- get (cache.py:52) — raises before any counter bump or move_to_end.
- put (cache.py:67) — raises before any eviction, popitem, counter bump, or store write. Critically, the TypeError cannot fire mid-eviction (which would drop a victim then fail to insert).
- peek (cache.py:82), remove (cache.py:92), __contains__ (cache.py:115) — raise before any mutation.

The TypeError propagates unwrapped (spec R15 / D5; CONSTITUTION 3 "raise at the boundary they originate from"). No catch-and-swallow, no silent coercion (CONSTITUTION 2.3 satisfied). No adversarial-key DoS vector beyond what the caller's own hashing exposes; hash collisions degrade dict performance generically and are out of scope for an in-process primitive (CPython SipHash randomization already mitigates the classic hash-flooding DoS for str/bytes keys).

### C3 — Reference retention / unbounded growth
Result: SAFE. Eviction (popitem), remove (del), and clear (OrderedDict.clear) all drop the cache reference to evicted key/value objects, so they become collectable once the caller also releases them. No auxiliary structure retains stale references (the design is a single OrderedDict, no shadow index, no listener list). The monotonic stats counters retain only integers, not key/value objects.

### C4 — Stats-counter overflow / abuse
Result: SAFE. _hits, _misses, _evictions are Python int (arbitrary precision) — they cannot overflow or wrap to a negative/attacker-controlled value the way a fixed-width counter could. They are read-only via @property (cache.py:123-146), so external code cannot write a poisoned value. Worst case is unbounded monotonic growth of a bigint, which is benign and bounded by total operation count; reset_stats (cache.py:105-113) lets a caller zero them. No exploitable abuse path.

## Findings

### F1 [LOW] Docstring does not warn that value memory is caller-bounded
- Location: src/tokenlab/cache.py:10-28 (class docstring)
- Class: Insecure-default / hardening best-practice (no concrete exploit)
- Description: capacity bounds entry count, not bytes. A caller that caches attacker-influenced large values (e.g. response bodies keyed by request) can be driven to high memory use within the entry limit. The behavior is correct per spec 3 (weight-based eviction is an explicit non-goal); this is purely a documentation-hardening nit so callers in a hostile-input context understand the boundary they own.
- Exploitation: Not directly exploitable in the library itself. Only matters at a downstream call site that stores untrusted, unbounded-size values; no such call site exists in this repo today.
- Remediation: Add one sentence to the class docstring: capacity limits entry count, not total memory; callers storing untrusted values should bound value size themselves. No code change.
- Disposition: LOW -> minor -> logged to BACKLOG.md per CONSTITUTION 12.1 (not an inline fix this iteration).
