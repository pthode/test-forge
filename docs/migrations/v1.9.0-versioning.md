# Migration guide: v1.8.0 ‚Üí v1.9.0 ‚Äî Release & versioning

This file tells you exactly what to change in your project's `CONSTITUTION.md` to align with the new ¬ß13 "Release & versioning" introduced in forge v1.9.0.

**Your `CONSTITUTION.md` is yours ‚Äî forge never auto-updates it.** Apply this migration when you're ready. Until you do, `release-engineer` will emit a CLARIFY (`BLOCKED: yes`) on its first Gate 1 invocation requesting that ¬ß13 be populated. The pipeline still runs ‚Äî only the release gate blocks. If you'd rather not adopt versioning, choose the `none` scheme in ¬ß13.1 below; the gate accepts that and skips version operations entirely.

---

## Step 1 ‚Äî Decide your scheme

Pick one of:

- **`semver`** ‚Äî `MAJOR.MINOR.PATCH`. Best for libraries, SDKs, CLIs, anything with external consumers who need to predict compatibility.
- **`calver`** ‚Äî `YYYY.MM.MICRO`. Best for apps, services, internal tools where users just want the latest. Communicates *when*, not *what changed*.
- **`none`** ‚Äî no formal versioning, releases tracked by git commit hash. Best for continuous-deployment services with no consumer-facing API.
- **`custom`** ‚Äî describe your own scheme in ¬ß13.2 manually. `release-engineer` will request clarification of bump rules on first run.

If you're unsure, default is `semver` for products that ship to others and `calver` for internal services.

```markdown
### 13.3 Changelog discipline

- `CHANGELOG.md` lives at the repo root and follows [Keep a Changelog](https://keepachangelog.com) format. Categories: `Added`, `Changed`, `Deprecated`, `Removed`, `Fixed`, `Security`.
- `doc-writer` appends every feature's user-visible changes to the `## [Unreleased]` section under the appropriate category. One bullet per change.
- `release-engineer` renames `Unreleased` to `[<new-version>] ‚Äî YYYY-MM-DD` on tag, opens a fresh empty `Unreleased` at the top.
- If the scheme is `none`, `CHANGELOG.md` is NOT maintained ‚Äî the git log is the history. `doc-writer` and `release-engineer` skip CHANGELOG operations.

### 13.4 Release process

- `release-engineer` (Gate 1) reads ¬ß13.1 and applies the scheme.
- Tags are immutable. Pre-release tags (`v1.2.0-rc.1`) are permitted; they do NOT close the `Unreleased` section.
```

### Per-scheme content for ¬ß13.2

**Semver:**

```markdown
- MAJOR ‚Äî breaking change to the public API. Existing consumers must update their code to keep working.
- MINOR ‚Äî new backwards-compatible capability. Consumers benefit but are not forced to change.
- PATCH ‚Äî bug fix or backwards-compatible correction. No new capability; behavior conforms more closely to the spec.
```

**Calver:**

```markdown
- `YYYY.MM` ‚Äî year and two-digit month of release. Always two digits for month (e.g. `2026.06`, not `2026.6`).
- MICRO ‚Äî sequential release counter within that month, starting at `1` (e.g. `2026.06.1`, `2026.06.2`).
- No semantic compatibility implication. Calver communicates _when_, not _what changed_.
```

**None:**

```markdown
- This project does not number releases. Deployments are tracked by git commit hash. `CHANGELOG.md` is not maintained. Re-enabling versioning requires a ¬ß11 amendment.
```

**Custom:**

```markdown
_Describe your scheme here. Include: what each component means, what triggers a bump of each, how pre-release tags work, and where the version field is stored._
```

## Step 4 ‚Äî Optional: backfill existing releases

If your project has already shipped versions before v1.9.0, you can backfill `CHANGELOG.md` from your git tag history. This is purely audit-trail work ‚Äî no agent depends on it.

```bash
git tag --sort=-creatordate
```

For each existing tag, look up the merged PRs / commits between it and the previous tag, summarise the user-visible changes, and add a section `## [<tag>] ‚Äî <tag-date>` to `CHANGELOG.md`. Or just add `## [<earliest-tag>] ‚Äî <date>` with a one-line "Initial release" and leave older history to git.
