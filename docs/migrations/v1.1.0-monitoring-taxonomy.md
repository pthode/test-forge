# Migration guide: v1.0.0 → v1.1.0 — Monitoring taxonomy

This file tells you exactly what to change in your project's `CONSTITUTION.md` to align with
the new §8 four-concern monitoring structure introduced in forge v1.1.0.

**Your `CONSTITUTION.md` is yours — forge never auto-updates it.** Apply these changes when
you're ready. Projects that don't apply them will continue working; the new monitoring
features (spec §10, observability-auditor Step 0) will fall back to inferring events from §7.

---

## What to replace

Find the section in your `CONSTITUTION.md` that covers observability/event logging. It will
look roughly like this (the original monolithic §8 from forge v1.0.0):

```
## <N>. Observability & event logging

- Every business-meaningful state change emits one structured log line. ...
- Log lines are machine-parseable ...
- Stable event names ...
- No PII or secrets in log payloads. ...
- Correlation IDs propagate across services. ...
- Log levels are calibrated: ...
- Every spec-mandated success criterion and every spec-mandated failure mode has a corresponding logged event. ...
- Metrics complement, never replace, structured events. ...
- Alerts have runbooks. ...
```

Replace the entire section (everything from the `##` heading to the end of the last bullet)
with the new four-sub-section structure below.

---

## Replacement content

Copy this block and adjust:
- Replace `<N>` with your constitution's section number.
- For each of §8b/§8c/§8d: choose `[scope: disabled]` or `[scope: enabled]` based on your project's needs.
- For enabled sections: fill in the TBD platform/retention fields. TBD is acceptable until you pick a platform.

```markdown
## <N>. Monitoring & observability

Monitoring covers four distinct concerns for this project, each with a scope marker agents use
to know what they must implement and audit.

---

### <N>a. Operational observability `[scope: always-on]`

- **Every business-meaningful state change emits one structured log line.** Schema:
  `{ts, level, event, actor_id, resource_id, ...domain_fields, trace_id}`.
- **Log lines are machine-parseable** (JSON or platform structured-log format). No `printf`
  debugging in production paths.
- **Stable event names** live in a code-level enum/const, not as inline strings.
- **No PII or secrets in log payloads.** Reference resources by stable ID; redact email,
  name, address, tokens, full request bodies.
- **Correlation IDs propagate across services.** Every inbound request creates or extracts a
  `trace_id`; every log line in the request path includes it.
- **Log levels are calibrated:** `ERROR` (request failed, oncall-worthy), `WARN` (degraded,
  recoverable), `INFO` (significant business events), `DEBUG` (disabled in production by default).
- **Every spec-mandated success criterion and every spec-mandated failure mode has a
  corresponding logged event.** `observability-auditor` enforces this mapping.
- **Metrics complement, never replace, structured events.**
- **Alerts have runbooks.** An alert without a linked runbook is noise.

> Add any project-specific operational observability rules here (e.g. "Brother dialogue content
> is never logged in plaintext").

**Agent responsibilities:** `spec-architect` enumerates required events in spec §10.
`developer` implements them from code-level enum/const. `devops-engineer` registers alert
rules and runbooks. `observability-auditor` verifies event coverage in Phase 3.
`release-engineer` Gate 5 confirms events are live in the target environment.

---

### <N>b. Audit logging `[scope: disabled]`

<!-- Change to [scope: enabled] if your project has compliance/legal obligations. -->
<!-- If enabled, fill in the TBD fields below. -->

Immutable, append-only record of privileged actions for compliance, legal, and accountability
purposes.

- **Retention:** TBD.
- **Platform:** TBD (append-only database table | CloudTrail | dedicated audit service).
- **Trigger events:** admin action, permission grant/revoke, data export, login/logout,
  record deletion.
- **Required fields:** `{ts, actor_id, action, resource_type, resource_id, ip, user_agent, result}`.
- **Immutability guarantee:** TBD.

> Audit logs are a separate concern from operational logs. They answer "who did what, when,
> to which resource?" for human accountability — not "what broke and why?" for debugging.

**Agent responsibilities when scope: enabled:** `spec-architect` must enumerate audit events
in spec §10. `database-designer` creates the audit table (append-only). `developer` writes to
it. `observability-auditor` verifies audit event coverage. `security-auditor` verifies
immutability and access controls.

---

### <N>c. Product analytics `[scope: disabled]`

<!-- Change to [scope: enabled] if your project needs user behaviour tracking. -->
<!-- If enabled, fill in the TBD fields below. -->

User behaviour and feature usage tracking for product insight, funnels, and experimentation.

- **Platform:** TBD (PostHog | Mixpanel | Amplitude | Segment | custom).
- **PII rules:** TBD (anonymous IDs only | pseudonymised | explicit consent required).
- **Client-side / server-side split:** TBD.

> Do not conflate analytics events with operational logs. Operational logs should not be
> exported wholesale to analytics platforms (PII risk), and analytics events should not
> substitute for operational coverage.

**Agent responsibilities when scope: enabled:** `spec-architect` enumerates analytics events
in spec §10. `developer` instruments them using the chosen platform's SDK.
`observability-auditor` verifies analytics events fire in staging.

---

### <N>d. Security event monitoring `[scope: disabled]`

<!-- Change to [scope: enabled] if your project needs anomaly detection / SecOps visibility. -->
<!-- If enabled, fill in the TBD fields below. -->

Anomaly detection and threat visibility for SecOps and incident response.

- **Platform:** TBD (Wazuh | Falco | GuardDuty | Datadog Security | SIEM of choice).
- **Trigger events:** failed-auth spike, impossible travel, privilege escalation, unusual
  data-volume export, scanner/fuzzer signatures.
- **Alert escalation path:** TBD.
- **Retention:** TBD (often 1 year minimum for compliance).

> Security monitoring events overlap with operational logs but have a different consumer
> (SecOps, not on-call engineering) and different retention. Route to the SIEM separately.

**Agent responsibilities when scope: enabled:** `security-auditor` includes security-event
coverage checks in every audit. `devops-engineer` provisions detection rules and alert
routing. `observability-auditor` spot-checks that trigger events fire in staging and are
reaching the SIEM.
```
