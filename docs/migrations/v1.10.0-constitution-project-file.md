# Migration guide: v1.9.0 → v1.10.0 — Project constitution file

This release introduces `CONSTITUTION.project.md` as a separate, project-owned file for project-specific constitution sections. It solves a collision problem: as forge grows numerically (§14, §15, …), it would have collided with projects that had added their own §14, §15 in `CONSTITUTION.md`.

The split mirrors the established `CLAUDE.md` / `CLAUDE.project.md` pattern. The forge file is the base; the project file extends it. Numbering is permanently disambiguated: forge uses §1, §2, …; project uses §P1, §P2, … .

**Your `CONSTITUTION.md` is yours — forge never auto-updates it.** Apply this migration when you're ready. Existing projects without a `CONSTITUTION.project.md` continue to work — agents that don't find the file proceed without it.

---

## Step 1 — Identify project sections to move

Open your `CONSTITUTION.md` and find any section you added beyond what forge ships. Forge owns §1–§13 (Stack, Non-negotiables, Code style, Test discipline, Security posture, Performance budgets, Accessibility, Monitoring, Definition of Done, Branching, Amendment, Backlog discipline, Release & versioning) plus the Revision log. If you have additional sections (typically numbered §14, §15, … in older projects), those are project extensions and need to move.

Example — a project might have:

```markdown
## 14. Data residency

All production data MUST remain within the EU. …

## 15. Customer-tenant isolation

Multi-tenant queries must include a tenant filter. …
```

---

## Step 2 — Create `CONSTITUTION.project.md`

At your project root, create `CONSTITUTION.project.md` with this content (substituting `<product-name>`):

```markdown
# <product-name> — Project-specific constitution extensions

This file extends `CONSTITUTION.md` with invariants that are specific to this project. Per `CONSTITUTION.md §11.2`:

- Numbering uses the `§P` prefix: `§P1`, `§P2`, `§P3`, … — never numerical, so future forge sections (§14, §15, …) cannot collide.
- **Additive only.** Project sections must NOT contradict or override any forge section in `CONSTITUTION.md` (§1–§13). If you need to change a forge section, go through §11.1 amendment process.
- Every agent that reads `CONSTITUTION.md` also reads this file. Project rules are binding in the same way forge rules are.
- `/forge-update` never touches this file. It is owned by the project team.

## How to add a project section

1. Pick the next sequential `§P<N>`.
2. Write the section as you would a forge section: title, rationale, bulleted rules.
3. Mention `§P<N>` in any spec, ticket, or code comment that depends on it.

## Project sections

<!-- Move your existing project-specific sections here, renumbered §P1, §P2, … -->
```

---

## Step 3 — Move and renumber

For each project-extension section in `CONSTITUTION.md`:

1. Copy the entire section (header + body) into `CONSTITUTION.project.md` under "Project sections".
2. Rename the header from `## 14. <title>` to `## §P1. <title>` (use sequential P-numbers starting from 1, regardless of what number the section had in CONSTITUTION.md).
3. Delete the original section from `CONSTITUTION.md`.

Continuing the example:

```markdown
## §P1. Data residency

All production data MUST remain within the EU. …

## §P2. Customer-tenant isolation

Multi-tenant queries must include a tenant filter. …
```

---

## Step 4 — Update cross-references

Search your codebase, specs, tickets, and docs for any reference to the old section numbers:

```bash
grep -rn "§14\|§15\|section 14\|section 15" .
```

Replace with the new `§P1` / `§P2` references. Most references will be in:

- `/docs/specs/*.md` (spec references to constitution)
- `/docs/qa-reports/*.md` (review reports citing violations)
- Code comments (rare but possible)
- `CLAUDE.project.md` (custom routing or anti-patterns)

---

## Step 5 — Verify forge sections are clean

After moving, `CONSTITUTION.md` should ONLY contain forge sections §1–§13 plus the Revision log at the end. Confirm:

```bash
grep -E "^## [0-9]+" CONSTITUTION.md
```

Should list §1 through §13 and nothing higher.

If you have a §14 or §15 left in `CONSTITUTION.md`, you missed it in Step 3 — go back and finish the migration.

---

## Step 6 — Verify the dual-file rule works

Once both files are in place, the next pipeline run should respect both. Quick sanity check: invoke any reviewer agent (`qa-reviewer`, `code-reviewer`, etc.) on a piece of code that would violate a project rule. The agent should flag the violation citing the `§P<N>` reference, not silently ignore it.

---

## Verify checklist

- [ ] `CONSTITUTION.md` contains only forge-owned sections §1–§13 plus Revision log.
- [ ] `CONSTITUTION.project.md` exists at the repo root with §P1, §P2, … filled in.
- [ ] No `grep` matches for `§14`, `§15`, etc. anywhere in the repo (or all remaining matches are intentional — e.g. the migration guide itself).
- [ ] A reviewer agent run cites `§P<N>` for any project-rule violation it finds.
- [ ] `/forge-update` does not modify either constitution file (it is excluded from the "will update" list).
