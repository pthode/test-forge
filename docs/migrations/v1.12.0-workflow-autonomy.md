# Migration: v1.12.0 — Workflow autonomy (CONSTITUTION §14)

**Your `CONSTITUTION.md` is yours — forge never auto-updates it.** Apply this migration when you're ready. Until you do, your project has no §14, and the orchestrator behaves as `review-all` (the pre-1.12.0 default: nothing lands without a human `ok`). Nothing breaks; you simply don't get the configurable autonomy until you opt in.

## What v1.12.0 adds

A new forge-owned section **§14 "Workflow autonomy"** that declares how much of the `commit → push → PR → merge` path the orchestrator drives on its own versus pausing for your review. It is layered, mirroring the permission model:

- **Floor (§14.1)** — never lowered by any setting: protected branches (`main` / `release/*`) are PR-only, the permission model is never loosened, CI must be green before merge, security-sensitive changes always get explicit human confirmation, and `URGENT` always surfaces.
- **Level (§14.2/§14.3)** — `review-all` (pause before every commit/PR/merge — the default), `review-critical` (autonomous on routine work, pause on critical changes), or `autonomous` (drive the whole path within the floor).
- **Critical (§14.4)** — security-sensitive set, schema migrations, public API contracts, new direct dependencies.
- **Personal override (§14.5)** — per-machine, may only be *stricter* than the project level.

The orchestrator (the `CLAUDE.md` router) reads §14 and gates accordingly; `/autopilot` Phase 4 uses it to decide whether to merge the PR or leave it open.

## Steps

1. **After `/forge-update`,** confirm the forge-owned files updated: `CLAUDE.md` has a "Workflow autonomy (CONSTITUTION §14)" section, `.claude/commands/init-project.md` has Step 17 (autonomy question), and `.claude/commands/autopilot.md` has Phase 4 §4f.

2. **Add §14 to your `CONSTITUTION.md`** (optional but recommended). Copy the `## 14. Workflow autonomy` section from `CONSTITUTION.template.md` (recover it from the forge source or `git show forge-upstream/main:CONSTITUTION.template.md`) and paste it just above your `## Revision log`.

3. **Choose your level.** Set §14.2 `Level` to one of `review-all` / `review-critical` / `autonomous`, and change `[scope: TBD]` to `[scope: set]`. If unsure, leave `review-all` — it matches how the pipeline behaved before this version.

4. **(Optional) Record a personal override.** If you personally want *more* review than the project level, note it as a per-machine `feedback` memory. It can only be stricter, never looser.

5. **Update your `CONSTITUTION.md` numbering references if you mirrored old text.** Anywhere your project copied the phrase "§1–§13 are forge-owned" or "may add §14, §15, …", update to "§1–§14" and "§15, §16, …" — §14 is now occupied by the forge.

## Verify

- [ ] `.forge-version` reads `1.12.0`.
- [ ] `CONSTITUTION.md` has a §14 (or you've consciously kept the `review-all` default with no §14).
- [ ] During a run, the orchestrator's commit/PR/merge behavior matches your chosen level — e.g. at `review-all` it pauses before merging; at `autonomous` it merges within the floor without asking.
- [ ] The floor still holds at every level: no direct push to `main`, security-sensitive changes still prompt.
