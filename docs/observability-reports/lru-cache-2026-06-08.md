## Event coverage matrix

### §8a — Operational observability

Per CONSTITUTION §8a, "every business-meaningful state change emits one structured log line." The specification and implementation argue that this requirement **does not apply** to an in-memory primitive. Per spec §10 (Observability plan):

> `LruCache` is a **pure in-memory primitive**: it performs no I/O, no persistence, and no network call; it owns no `actor_id`, `resource_id`, or `trace_id`; and all of its state changes (hit, miss, insert, update, eviction) are deterministic consequences of caller method calls, fully observable through `stats()` and the method return/raise contract.

The mapping rule (§8a) states: **Every spec-mandated success criterion and every spec-mandated failure mode has a corresponding logged event.**

The spec §8 and §10 define the contract for this feature to emit **zero operational log events**. The mapping is satisfied vacuously: the success criteria (R1–R14 from spec §8) are asserted directly by unit tests, not by log inspection, and the implementation correctly emits no events. This is not an omission — it is the mandated correct behavior.

| Spec ref | Requirement | Success criterion | Logged event? | Rationale |
|----------|-----------|-----------|-----------|-----------|
| R1–R14 | Construction, insertion, lookup, eviction, statistics | All testable via method return and `stats()` contract | **None — by design** | Pure in-memory utility; state changes are fully observable to the caller via `stats()` interface. Emitting logs from within the primitive would duplicate the caller's boundary logging and attach no correlation context the primitive owns. Per spec §10 and ticket §4/§11, §8a does not apply. |

## PII / secret hygiene analysis

**Risk assessment:** NONE.

The cache performs no I/O, no serialization, no logging, and no external communication. It accepts only caller-supplied hashable keys and arbitrary values. There is zero logging infrastructure in the implementation and zero risk of logging PII or secrets from within the primitive. The caller is responsible for filtering sensitive data if needed, but that is the caller's logging discipline, not the cache's.

## Metrics & alerts

**Applicable:** NO.

The cache is a pure utility with no network boundary, no external service calls, no async work, and no distributed coordination. An alert rule requires a failure mode that is (a) externally observable and (b) requires human intervention. Neither applies here. The `stats()` interface exposes `hits`, `misses`, `evictions`, and `size` — the caller's own application logic is the right place to instrument cache performance metrics and alert on hit-rate thresholds, eviction frequency, etc. These are business-level observability concerns, not cache-internal ones.

## Findings

**None.** The specification correctly exempts this feature from §8a operational logging. The implementation correctly emits no events. The observability audit confirms the rationale is sound and the implementation is correct.
