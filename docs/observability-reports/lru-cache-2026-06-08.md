## Feature characterization

`LruCache` is a pure, single-instance, in-process memory cache utility. Characteristics:

- **No HTTP/network boundary** — in-process library class only.
- **No actor principal** — single-threaded, caller-owned instance.
- **No shared resource ID** — internal per-instance state.
- **No business-meaningful state change visible externally** — cache hits/misses/evictions are memoization mechanics internal to the caller's usage.
- **Hot O(1) path** — every get/put operation must be fast (§spec R10).

Per CONSTITUTION §8a, "business-meaningful" events are those "an auditor, a support engineer, or a product manager would want to reconstruct after the fact." Cache operations do not meet this threshold — they are implementation details of callers' own business logic.

## Spec §10 (Observability plan) review

The spec provides an explicit observability table in §10 documenting the two candidate events (eviction and invalid capacity) and **justified reasoning for why neither is logged by the library**:

| Spec ref | Event/Concern | Logging status | Rationale |
|----------|---|---|---|
| R6 (eviction) | Cache eviction occurs | **Not logged** | Exposed via `stats.evictions` (caller-pollable counter). Eviction is normal, expected, per-instance behavior; no process-global metric or alert applies. Logging from the hot path would violate §8a "no printf in production paths" and R10 performance intent. |
| R1 (invalid capacity) | Capacity <= 0 | **Not logged** | Surfaced as `ValueError` to caller at construction time. Caller's own observability layer records it if relevant. Library does not own a logger. |

## Event coverage matrix

| Spec ref | Event name | Implementation location | Level | Trace ID | PII risk | Status |
|----------|---|---|---|---|---|---|
| R1 (failure) | invalid capacity | /src/tokenlab/cache.py:18–19 `ValueError` | exception | n/a | no | Caller-facing, not library-logged |
| R6 (event) | eviction | /src/tokenlab/cache.py:41–42 `stats.evictions` counter | counter (polled) | n/a | no | Exposed via stats snapshot |

**Justification:** Both mapped concerns are intentionally **not logged** by the library:
1. Eviction is exposed as a counter in the public `stats` property; callers poll it if they need to track it.
2. Capacity validation raises `ValueError` immediately; the caller's logging layer is responsible for observing this error if relevant to their business logic.

This design is sound under §8a because:
- Neither event represents a business-meaningful state change visible outside the instance.
- Logging from the O(1) hot path would contradict both §8a's "no printf in production paths" and the spec's locked §R10 (O(1) amortized).
- The contract is explicit: evictions are countable via `stats`, capacity errors are exceptions.
